# Epoll事件模块详解

在 Linux 的 `epoll` 机制中，`epoll_ctl()` 函数用于控制 epoll 实例（添加、修改或删除监视的文件描述符）。其操作码（`op` 参数）定义了要执行的操作类型，共有以下三种：

### 1. **`EPOLL_CTL_ADD`** (`1`)

- **作用**：向 epoll 实例 **添加** 一个新的文件描述符进行监视。
- **参数要求**：
  - `fd`：需监视的文件描述符。
  - `event`：指向 `struct epoll_event` 的指针，指定关注的事件（如 `EPOLLIN`、`EPOLLOUT` 等）和用户数据。
- **错误情况**：若 `fd` 已在 epoll 中，会返回 `EEXIST` 错误。

### 2. **`EPOLL_CTL_MOD`** (`2`)

- **作用**：**修改** 已注册文件描述符的监视事件。
- **参数要求**：
  - `fd`：已在 epoll 中的文件描述符。
  - `event`：指向新的 `struct epoll_event` 结构体的指针，用于更新事件设置。
- **错误情况**：若 `fd` 未在 epoll 中，会返回 `ENOENT` 错误。

### 3. **`EPOLL_CTL_DEL`** (`3`)

- **作用**：从 epoll 实例中 **删除** 文件描述符的监视。
- **参数要求**：
  - `fd`：需删除的文件描述符。
  - `event`：可传入 `NULL`（推荐），或任意指针（内核会忽略此参数）。
- **错误情况**：若 `fd` 未在 epoll 中，会返回 `ENOENT` 错误。





###  以下是 Linux `epoll` 机制支持的所有核心事件标志及其详细说明，结合使用场景和注意事项整理而成：

---

### ⚙️ **一、核心 I/O 事件**
1. **`EPOLLIN`** (`0x001`)  
   - **触发条件**：关联的文件描述符有数据可读（接收缓冲区非空）。  
   - **典型场景**：  
     - 监听套接字收到新连接（`accept`）；  
     - 客户端套接字收到数据（`read`）；  
     - 对端正常关闭连接（需结合 `read` 返回 `0` 判断）。  

2. **`EPOLLOUT`** (`0x004`)  
   - **触发条件**：文件描述符可写（发送缓冲区有空间）。  
   - **使用注意**：  
     - 默认持续触发（LT 模式），ET 模式下仅在缓冲区从满变为空时触发一次；  
     - 需配合非阻塞 I/O 避免阻塞。  

3. **`EPOLLPRI`** (`0x002`)  
   - **触发条件**：有紧急数据（带外数据，OOB）到达（如 TCP `URG` 标志数据）。  
   - **场景**：高优先级消息（如 SSH 中断信号）。  

4. **`EPOLLERR`** (`0x008`)  
   - **触发条件**：文件描述符发生错误（如写入已关闭的 socket）。  
   - **关键特性**：  
     - 内核**自动监控**，无需显式设置；  
     - 通常与 `EPOLLHUP` 同时出现。  

5. **`EPOLLHUP`** (`0x010`)  
   - **触发条件**：连接被挂断（如本端调用 `shutdown` 或对端异常断开）。  
   - **注意**：与 `EPOLLRDHUP` 区别在于可能由本地操作触发。  

6. **`EPOLLRDHUP`** (`0x2000`, Linux 2.6.17+)  
   - **触发条件**：对端关闭连接（发送 `FIN` 包）或半关闭写端。  
   - **优势**：替代 `read` 返回 `0` 的检测方式，支持更精确的断开识别。  

---

### 🎛️ **二、高级控制标志**
1. **`EPOLLET`** (`1 << 31`)  
   - **作用**：设置为边缘触发（Edge Triggered）模式（默认水平触发 LT）。  
   - **ET 模式特性**：  
     - 仅在状态变化时触发（如缓冲区从空→非空）；  
     - **必须非阻塞 I/O + 循环读写至 `EAGAIN`**；  
     - 性能更高，但编程复杂度增加。  

2. **`EPOLLONESHOT`** (Linux 2.6.2+)  
   - **作用**：事件触发后自动禁用该描述符的监听，需通过 `EPOLL_CTL_MOD` 重新启用。  
   - **场景**：避免多线程中同一事件被重复处理。  

---

### 📊 **三、常见事件组合及含义**
| **事件组合**            | **含义**                                |
| ----------------------- | --------------------------------------- |
| `EPOLLIN`               | 数据可读或新连接到达（监听 socket）     |
| `EPOLLIN \| EPOLLRDHUP` | 对端关闭连接（正常 FIN 包）             |
| `EPOLLOUT`              | 可发送数据（缓冲区有空位）              |
| `EPOLLERR \| EPOLLHUP`  | 连接错误或挂断（需关闭描述符）          |
| `EPOLLIN \| EPOLLET`    | ET 模式下的可读事件（需一次性读完数据） |

---

### ⚠️ **四、注意事项**
1. **错误与挂断事件的监控**：  
   - `EPOLLERR` 和 `EPOLLHUP` **无需显式设置**，内核始终监控。  
2. **边缘触发模式的要求**：  
   - ET 模式必须使用非阻塞 I/O，并循环读写直到返回 `EAGAIN` 或 `EWOULDBLOCK`。  
3. **连接断开的检测**：  
   - 对端异常断开（如拔网线）**不会主动触发事件**，需通过写操作超时或 `EPOLLERR` 间接发现。  
4. **跨内核版本兼容性**：  
   - `EPOLLRDHUP` 需 Linux ≥ 2.6.17，旧内核需通过
   -  `read` 返回 `0` 判断对端关闭。  

通过合理组合这些事件，可高效实现高并发网络服务（如 Web 服务器、实时通信系统）。建议结合非阻塞 I/O 和 ET 模式以最大化性能。
